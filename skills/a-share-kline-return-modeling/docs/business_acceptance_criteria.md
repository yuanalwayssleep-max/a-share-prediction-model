# 业务验收标准

更新时间：2026-05-30 GMT+8

## 1. 文档目的

本文从 `user / UAT` 视角定义 A 股 5 日预测模型的业务目标、非目标、日常输出预期、验收标准和不可接受情况。

本文件不替代技术方案、开发计划、标签设计或测试计划。它只回答一个问题：

```text
作为业务使用者，我到底希望这个系统帮我完成什么？
```

## 2. 项目业务目标

项目目标是建设一个 A 股 5 日短线候选股辅助系统：

```text
每天基于历史日 K、个股特征和市场环境，筛选未来 5 个交易日更可能表现强势的股票候选，辅助用户形成观察清单和交易纪律。
```

更具体地说，系统应该帮助用户：

- 每天快速得到值得重点关注的候选股票。
- 理解这些候选为什么被选出来。
- 判断当前市场环境下是否适合正常出手、降低出手或谨慎观望。
- 复核模型过去在组合层面的表现，包括收益、回撤、弱月份和稳定性。
- 用固定流程生成结果，避免每次依赖人工或 agent 临时猜脚本。

## 3. 非目标

以下内容不属于本项目承诺范围：

- 不承诺稳定盈利。
- 不作为投资建议或自动交易系统。
- 不预测涨停板，也不追求单日神票命中。
- 不以单次回测收益漂亮作为业务成功标准。
- 不允许使用未来收益、未来标签或判卷字段生成最终执行信号。
- 不要求 `user` 角色承担开发测试、契约测试或代码质量验证，这些由 `qa` 负责。

## 4. 用户每天期望看到什么

理想状态下，每个交易日应能生成一份用户可读的最终信号输出，至少包含：

| 项目 | 说明 |
|---|---|
| 候选股票 | 默认 Top3，或经批准后的动态 TopN |
| 排名/分数 | 展示候选之间的相对强弱，不要求用户理解模型内部全部细节 |
| 选择理由 | 用简短语言解释主要排序依据、趋势/量价/市场环境等因素 |
| 风险提示 | 当前是否弱市、是否高波动、是否需要谨慎出手 |
| 操作纪律 | 观察窗口、持有周期、买卖时点、成本/滑点假设 |
| 回测摘要 | 当前主线在历史验证中的组合表现和主要风险 |
| 数据时间 | 明确输入数据截止时间，避免误以为包含未来信息 |

## 5. 业务验收标准

### 5.1 可用性

业务侧认为系统可用，需要满足：

- 有稳定入口能生成每日候选清单。
- 输出文件字段清晰，用户不用翻源码也能理解主要含义。
- 结果能解释为“辅助观察/决策纪律”，而不是只给一串 CSV 分数。
- 新 agent 根据 README、STATUS 和本文档可以理解项目目标。

### 5.2 有效性

业务侧认可模型方向，需要满足：

- M3 个股排序在 walk-forward 验证中显著优于随机或旧基线。
- Top3 结果不能只依赖少数强月份撑均值，需要展示弱月份表现。
- M4 组合级回测必须能说明资金曲线、最大回撤、正收益月份、暴露比例和交易次数。
- M5 仓位/出手策略只有在证明改善风险收益后，才能进入默认执行口径。

### 5.3 风险可控

业务侧接受系统辅助，需要满足：

- 最终信号链路 truth-free，不允许读取 `future_*`、`label_*` 或判卷字段。
- 弱市、强波动、过热、近涨停等风险应有提示或可选过滤。
- 出手数量和仓位规则必须能解释，不能因为实验收益好看就直接变复杂。
- 所有主线变更需要有 CR、Gate Review 或明确审批记录。

### 5.4 可复现

业务侧要求结果可复核，需要满足：

- 有明确依赖安装方式和主流程命令。
- 同一输入数据、同一配置下，输出结果应可重复生成。
- 关键报告有索引，能区分正式依据、候选依据、实验产物和历史产物。
- 会议结论、风险、决策和验收状态不能只散落在聊天里。

## 6. 不可接受情况

以下情况从业务视角不可接受：

- 最终信号依赖带 truth 的输入文件。
- 文档说 A，代码实际跑 B，新 agent 无法判断哪个是准的。
- 只展示平均收益，不展示回撤、亏损月份和随机对照。
- 频繁在同一验证窗口上调参，然后把结果当泛化能力。
- 每天输出需要人工猜脚本、补路径、修环境才能跑。
- 用户代表被迫判断 pytest、schema guard、单元测试等工程细节。

## 7. UAT 场景

### 场景 1：每日候选查看

```text
给定最新可用日 K 数据，系统生成最终候选清单。
用户可以在 3 分钟内理解：今天关注哪些股票、为什么、风险是什么。
```

验收点：候选清单存在、字段清楚、说明可读、无 future/truth 字段暴露。

### 场景 2：弱市判断

```text
当市场环境偏弱或模型历史弱月份类似时，系统能提示风险，而不是机械给出满仓信号。
```

验收点：有风险提示或保守策略说明；若仍建议 Top3/full_size，应解释依据。

### 场景 3：历史表现复核

```text
用户希望知道当前策略过去表现如何。
系统能提供组合级报告，包括收益、回撤、正收益月份、交易次数和暴露比例。
```

验收点：报告路径清楚，指标口径一致，不混用实验产物和正式依据。

### 场景 4：主线变更判断

```text
如果 rd 或 qa 提出更换模型、出手数量、仓位规则或标签设计，user 只判断业务价值和接受条件，不负责工程测试细节。
```

验收点：PM 给出影响说明，QA 给出测试证据，RD 给出技术依据，user 只做业务/UAT 判断。

## 8. 当前业务判断

截至 2026-05-30，业务侧可以接受的判断是：

```text
M3 个股排序方向具备阶段可用性。
M4 组合级回测已具备业务复核基础，但仍需正式验收收口。
M5 固定 full_size 暂作为执行基准，combined_size_v2 保留为候选风险开关。
M6/M7 已开始解决 truth-free 和测试固化问题，这是进入业务使用前的必要条件。
```

业务侧还不应把本项目视为“可直接实盘自动执行”的系统。当前更合理的定位是：

```text
研究验证中的 A 股短线候选与交易纪律辅助系统。
```

## 9. 角色边界

| 角色 | 业务验收中的职责 |
|---|---|
| pm | 管理目标、范围、Gate、CR、风险和验收节奏 |
| rd | 提供技术实现、模型依据和工程变更说明 |
| qa | 验证开发测试、回归测试、契约测试、复现性和质量门禁 |
| user | 判断业务目标是否清楚、输出是否有用、风险是否可接受、UAT 是否通过 |
| default | 汇总各方结论，向用户简洁汇报 |

## 10. 后续建议

- 将本文作为 `user` 角色参与小时报会议和 Gate Review 的默认依据。
- 后续新增最终信号日报时，优先对齐本文第 4 节的用户输出预期。
- 主线策略发生变化时，PM 应检查本文是否需要更新；如果涉及业务目标变化，需要用户确认。
