# 变更请求规范

## 目的

开发阶段如果需要修改已经确认过的项目方案、开发计划、阶段目标、验收口径或主线实现，必须先创建变更请求文档，等待审批后再实施。

这条规则用于避免：

```text
边做边改目标
回测口径漂移
验收标准被结果反向影响
未经确认就改开发计划
```

## 适用范围

需要提交变更请求的情况：

```text
修改里程碑目标
修改阶段验收标准
修改标签定义
修改交易或回测口径
修改模型主线
修改候选池规模或 TopK 目标
修改特征白名单/黑名单
新增或删除核心脚本
改变输出文件结构
改变 walk-forward / 测试集规则
把实验方案正式并入主流程
替换当前主模型或最终信号链路
```

不需要提交变更请求的情况：

```text
修复明确 bug，且不改变业务口径
补充注释或文档错别字
增加不影响主流程的日志
运行已有脚本生成报告
格式化代码
临时实验脚本
离线对照实验
诊断分析报告
不进入主流程的模型、特征、权重试错
outputs 下的研究性实验结果
```

判断原则：

```text
实验可以直接做。
主线变更才走 CR。
如果实验结果要进入项目方案、开发计划或主流程，再提变更请求。
```

## 流程

```text
1. 发现变更需求
2. 新建变更请求文档
3. 状态标记为 待审批
4. 用户审批或产品经理评审
5. 审批结论直接写入对应变更请求文件
6. 审批通过后实施
7. 实施完成后回填结果
8. 如未通过，保持原方案不变
```

## 审批记录要求

后续评审变更请求时，审批结论必须直接写入对应 CR 文档的“审批记录”部分，而不是只在聊天中口头说明。

审批记录至少包含：

```text
审批结论：已批准 / 已拒绝 / 暂缓批准 / 补充后再审
审批时间
审批人
产品经理评审人，如适用
审批意见
批准范围或拒绝原因
约束条件
后续要求
```

除非用户明确要求，否则审批记录写入源 CR 文档即可，不额外创建备份文件，不要求 commit/push。

## GitHub 同步要求

变更请求不要求每次创建、更新、审批或关闭都提交并推送到 GitHub。

以下情况再进行 GitHub 同步：

```text
用户明确要求“上传 GitHub / 提交 / push”
某个阶段实现完成，需要归档稳定版本
需要在远端保留评审版本快照
```

本地变更请求文档可以正式进入审批流程；是否推送 GitHub 不影响审批有效性。

## 状态

```text
待审批：已提出，未批准，不得实施
已批准：可以实施
已拒绝：不得实施
已实施：已按批准内容完成
已撤销：提出方取消
```

## 文档命名

变更请求放在：

```text
skills/a-share-kline-return-modeling/docs/change_requests/
```

命名格式：

```text
CR-YYYYMMDD-NNN-简短标题.md
```

示例：

```text
CR-20260529-001-adjust-m3-target-to-top50-recall-rerank.md
```

## 模板

```text
# CR-YYYYMMDD-NNN 标题

状态：待审批
提出日期：YYYY-MM-DD
提出人：Codex
影响阶段：M?

## 变更摘要

一句话说明要改什么。

## 背景

为什么提出变更。

## 当前方案

原来的目标、口径或实现方式。

## 建议变更

新的目标、口径或实现方式。

## 影响范围

影响哪些文档、脚本、输出、验收标准。

## 预期收益

为什么值得改。

## 风险

可能带来的问题。

## 验收标准

变更通过后如何判断成功。

## 审批记录

待用户填写。

## 实施记录

审批通过后填写。
```
