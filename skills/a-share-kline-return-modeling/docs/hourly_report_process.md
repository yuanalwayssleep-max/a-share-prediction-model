# 项目小时报规范

更新时间：2026-05-30 GMT+8

## 目的

项目小时报用于记录每个工作时段的项目状态、关键结论、风险变化和下一步动作，避免多 agent 协作时出现：

```text
只在聊天里说过，文档里没留下
实验越跑越多，项目状态没人收口
风险被发现但没有 owner
阶段 gate 结论散落在对话和输出文件里
```

小时报不是流水账，也不是替代 CR。它是 PM/PMO 对项目进展的短周期状态快照。

## 适用范围

需要写小时报的情况：

```text
完成一次 PM 进度检查
完成一次阶段 gate review
完成一次关键实验或回测
发现新的阻塞、风险或范围变化
多 agent 协作完成一个阶段性动作
用户要求“记录进度 / 存到小时报 / 做小时报”
```

不需要写小时报的情况：

```text
纯粹修正文档错别字
单次小命令检查且没有形成项目结论
临时探索但没有影响状态、风险或计划
```

## 存放位置

小时报统一放在：

```text
skills/a-share-kline-return-modeling/docs/hourly_reports/
```

命名格式：

```text
YYYY-MM-DD-HHMM-简短主题.md
```

示例：

```text
2026-05-30-0856-pm-progress-review.md
```

## 角色职责

```text
default：接收用户要求，触发小时报，向用户汇总关键结论。
pm：负责小时报主内容，包括状态、WBS、风险、gate 结论和下一步。
rd：提供技术证据、脚本输出、测试结果、变更文件清单。
user：提供 UAT/用户视角验收反馈。
```

## 小时报模板

```text
# 小时报：主题

记录时间：YYYY-MM-DD HH:mm GMT+8
记录人：pm
项目阶段：M?
报告类型：进度检查 / Gate Review / 实验总结 / 风险更新 / 交付收口

## 1. 本小时目标

本时段要解决或确认什么。

## 2. 输入材料

- 检查的文件、报告、脚本、输出或对话来源。
- 如涉及命令，仅记录非敏感摘要，不贴 secrets、.env、完整日志。

## 3. 当前结论

一句话说明项目当前状态。

## 4. WBS 进度

| 工作包 | 状态 | 证据/说明 | 下一步 |
|---|---|---|---|
| M0 | 未开始 / 进行中 / 已完成 / 有条件通过 |  |  |

## 5. 风险与阻塞

| 风险/阻塞 | 等级 | 影响 | Owner | 建议动作 |
|---|---|---|---|---|

## 6. Gate 结论

```text
建议结论：通过 / 有条件通过 / 暂缓 / 不通过
通过条件：
下一阶段准入条件：
```

## 7. 决策与变更

- 已确认决策：
- 待确认决策：
- 是否需要 CR：是 / 否
- CR 编号，如有：

## 8. 下一小时/下一阶段动作

1. 动作一
2. 动作二
3. 动作三

## 9. 备注

其他需要保留的上下文。
```

## 质量要求

- 每份小时报必须有明确状态结论，不允许只贴过程。
- 每个高风险项必须有 Owner 和建议动作。
- Gate Review 必须写“通过 / 有条件通过 / 暂缓 / 不通过”。
- 若发现主线变更、验收口径变化或回测口径变化，必须标记是否需要 CR。
- 不得记录 `.env`、token、密钥、完整敏感日志。
- 若小时报引用实验结果，必须写明对应报告、脚本或输出文件路径。

## 与 CR 的关系

小时报可以记录“发现了变更需求”，但不能替代 CR 审批。

```text
小时报：记录状态、发现、建议、风险。
CR：审批主线、目标、验收口径、交易口径或核心实现变更。
```

如果小时报结论涉及主线变更，PM 必须在小时报中标记“需要 CR”，并创建或引用对应 CR 文档。
